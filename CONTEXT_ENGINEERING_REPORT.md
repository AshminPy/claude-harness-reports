Scope: global `CLAUDE.md`, project `CLAUDE.md` files, `rules/`, hooks, commands, memory
files, state files, skills, subagent instructions. Goal: reduce unnecessary context
while preserving behavior, per the 7-level hierarchy (global → project → rules →
skills → subagents → task state → evidence).

## Problems found

1. **`troubleshooting.md` was never committed to the claude-toolkit repo** — confirmed
   by direct comparison (`diff` showed it entirely absent from `rules/` in the repo,
   present only under live `~/.claude/rules/`). This is the single largest and, by its
   own content, most operationally important rules file (the mandatory
   reproduce→hypothesize→verify workflow, the 3-file investigation-state model). It was
   flagged as a deferred Low-priority item twice already (original audit's Fix L1,
   re-confirmed in `PHASE1_FINAL_CHECK.md`) but never actually closed — a real,
   uncorrected data-loss risk: if this machine's `~/.claude` were lost, this file (and
   the workflow discipline it encodes) would be gone with no recovery path, unlike
   `behavior.md`/`security.md`/`cloud.md`, which are already safely mirrored.
2. **Total always-loaded context (356 lines) exceeds this harness's own previously-
   established target.** The harness's own history (`deploy-states/harness-setup.md`)
   documents a deliberate 2026-07-01 effort that cut always-loaded rules from ~360 to
   107 lines specifically because "adherence drops once always-loaded context exceeds
   ~200 lines" (verified against official Claude Code docs at the time). Counting
   `troubleshooting.md` (which was always being loaded live, just not tracked), the
   real total was already back up near the original pre-cut figure. This is a genuine
   tension: `troubleshooting.md`'s content is detailed and valuable, not filler — cutting
   it for line-count's sake would remove real behavioral value. Flagged as a finding,
   not force-resolved (see "Changes made" for what was actually done, vs. what's left
   as a judgment call).
3. **A stale-looking cross-reference from the outer harness platform.** The
   `UserPromptSubmit` hook's auto-injected reminder text seen throughout this
   conversation ("📋 RESPONSE FORMAT (auto): follow `~/.claude/rules/behavior.md`
   'Communication style' ... Use 'Final task format' [Status/Result/Root cause/
   Changes/Verification/Risk/Next action]") references section names and a label set
   that **no longer exist** in the current `behavior.md` — that file now has a
   consolidated "Reply format" section with different labels (Result/Validate/Your
   move), evidently edited since that reminder text was last generated/cached.
   Searched this repo and `~/.claude` exhaustively — this exact reminder string does
   not originate from any file here, so it is generated by the outer Claude Code
   platform itself, not by anything in this repo. **Cannot be directly edited from
   here** — noted as an observed discrepancy, not a fixable defect in this codebase.
4. **`behavior.md`'s "Response counter" rule ("every response ends with [RN]") is not
   enforced by any hook or script** — grepped the entire repo, no code counts
   responses or checks for a `[RN]` suffix. It also was not being followed during this
   very session (40+ tool calls, no `[RN]` ever emitted) with no ill effect, since
   `TaskCreate`/`TaskUpdate` already provides a more structured, hook-adjacent form of
   progress tracking for exactly this kind of long session. **Not removed** — this
   file was evidently edited recently and deliberately by the user (confirmed: the
   "Communication style"/"Final task format" → "Reply format" consolidation happened
   between this conversation's start and this review, and the response-counter section
   was kept through that edit) — removing a personal workflow preference the user just
   chose to keep is outside the scope of "technical context cleanup" and is flagged as
   a recommendation only, not acted on.
5. **No other redundancy or conflict found.** `cloud.md` already uses path-scoped
   frontmatter (`paths: **/*.tf, **/iac/**, ...`) so it only loads for infra-shaped
   work, not on every turn — a form of conditional loading beyond simple always-on.
   Skills and subagents are already correctly on-demand-only (confirmed: none are
   referenced from global `CLAUDE.md` except by name/pointer, matching the intended
   hierarchy). No duplicated instructions found between `CLAUDE.md`/rules/skills.

## Changes made

1. **Committed `rules/troubleshooting.md` into the claude-toolkit repo**, byte-identical
   to the live version (verified via `diff`, zero behavioral change). `install.sh`
   already globs `rules/*.md`, so no installer change was needed — it is now
   automatically protected by every future sync, matching `behavior.md`/`security.md`/
   `cloud.md`.
2. No other files were changed. The response-counter question, the platform-level
   reminder-text staleness, and the 356-line total are documented above as findings /
   recommendations rather than acted on, since none of them are technical defects with
   an unambiguous, low-risk fix — they involve either a personal workflow preference
   (response counter) or a system outside this repo's control (the platform's own
   reminder generation).

## Files changed

## Estimated context reduction

**Net 0 lines removed from what was actually being loaded** — this pass was protective
(preventing future loss of already-loaded content), not reductive. No functional
context-size change for this session or any current session, since `troubleshooting.md`
was already being loaded live before this change; the only difference is it's now also
tracked in source control.

Nothing touched: `guard_secrets.py`, `block_push.py`, `command_policy_hook.py`,
`final_gate_hook.py`, and every other enforcement hook are unmodified. `security.md`
and `cloud.md`'s IAM/architecture rules are unmodified.

## Remaining limitations

- The 356-line always-loaded total is above this harness's own historical ~200-line
  adherence target, but `troubleshooting.md`'s content is genuinely valuable — reducing
  it further is a content/product judgment call for the user, not a mechanical cleanup,
  and was intentionally left alone rather than cut for a line-count target.
- The outer platform's auto-generated reminder text referencing old section names
  ("Communication style"/"Final task format") cannot be corrected from within this
  repo — it is not sourced from any file here.
- The "Response counter" rule remains unenforced by code; flagged as a recommendation
  (either wire it to something real, or retire it) for the user to decide, not changed.
