# Harness Guide

**Canonical reference for maintaining this harness going forward.** Where this
document and an older report disagree, this document wins — it consolidates and
supersedes the scattered explanations in `HARNESS_OPERATIONS.md`,
`CONTEXT_ENGINEERING_REPORT.md`, `SKILLS_IMPLEMENTATION_REPORT.md`, and the various
`PHASE*`/`HARNESS_RELIABILITY_*` reports. Those reports remain as historical record of
*how* each piece was built and verified; this doc is the one place to read to
understand and maintain the harness *as it stands today*.

---

## 1. Architecture overview

```
┌─────────────────────────────────────────────────────────────┐
│  ~/projects/ai-projects/claude-toolkit   (SOURCE OF TRUTH)   │
│  hooks/  scripts/  skills/  agents/  rules/  policies/       │
│  mistakes/  templates/  .claude/commands/                    │
└───────────────────────────┬───────────────────────────────────┘
                            │ install.sh (idempotent copy)
                            ▼
┌─────────────────────────────────────────────────────────────┐
│  ~/.claude                                (LIVE MIRROR)      │
│  Every Claude Code session on this machine runs from here.   │
│  settings.json wires hooks; skills/agents load on demand.     │
└───────────────────────────┬───────────────────────────────────┘
                            │ per project, at runtime
                            ▼
┌─────────────────────────────────────────────────────────────┐
│  <project>/.claude/                        (PER-PROJECT STATE)│
│  validation-state.json, state/task-queue.md,                  │
│  state/evidence-ledger.json, task-contract.md                 │
└─────────────────────────────────────────────────────────────┘
```

Three layers, three different lifecycles:
- **Source repo** — versioned, reviewed via PR, the only place you should edit.
- **Live mirror** — a plain copy, regenerated by `install.sh`, never hand-edited.
- **Per-project state** — created and updated automatically by hooks/skills while you
  work in a given project; not part of the harness's own source at all.

## 2. How the harness works

Two enforcement philosophies, deliberately separated:

- **Enforcement lives in hooks/scripts (real code, cannot be talked out of it).**
  Destructive-command blocking, retry/circuit-breaker limits, evidence-trust checks,
  and stale-state cleanup are all executable Python, covered by
  `scripts/test_reliability_fixes.py` (36 tests) and `scripts/command_policy.py --test`
  (143 tests).
- **Knowledge lives in skills/agents (prose, loaded only when relevant).** How to run
  a deploy pipeline, how to review Terraform, how to write a runbook — all prompt-driven,
  not hook-enforced, and loaded on demand so they don't cost context on unrelated turns.

### The hook lifecycle (what fires, in order, for a typical turn)

1. **SessionStart** (`session_start.sh`) — detects the project, injects routing
   context, checks for unfinished task-queue state.
2. **UserPromptSubmit** (`prompt_submit.py`) — on every message: logs to the audit
   log, auto-sets `risk_mode=STRICT` on infra/security keyword match, injects
   mistake-memory matches, injects the anti-sycophancy reminder on pushback phrases,
   handles Learning Mode / Away Mode / Project Autopilot Mode triggers.
3. **PreToolUse**:
   - `guard_secrets.py` (Read/Edit/Write/NotebookEdit) — hard-blocks known
     secret-file patterns.
   - `block_push.py` (Bash) — hard-blocks `git push` to `main`/`master`/`init`.
   - `check_commit.py` (Bash) — blocks AI-attribution text in commit messages.
   - `command_policy_hook.py` (Bash) — classifies every command
     (`allowed`/`blocked_needs_privileged`/`blocked_secret`/`blocked_hard`), runs the
     retry/circuit-breaker ladder, auto-logs command-level evidence.
   - `autopilot_scope_hook.py` (Edit/Write/MultiEdit) — when Project Autopilot Mode is
     active, auto-allows in-scope file edits without prompting.
4. **PostToolUse**:
   - `bash_output_filter.py` — truncates/redacts large Bash output, records the real
     post-execution outcome (exit code, bounded stdout) as trusted evidence.
   - `changed_files_fence_hook.py` — flags file changes outside the declared scope.
   - `auto_sync_claude_md.py` — mirrors per-project `CLAUDE.md` edits to a backup.
5. **PostToolUseFailure** (`post_tool_failure.py`) — logs the failure, records it into
   the same retry/attempts log as the PreToolUse path.
6. **PreCompact / PostCompact** — preserve and re-inject key rules/state across a
   context compaction.
7. **Stop** (`final_gate_hook.py`, `stop_session.sh`) — for STRICT-mode tasks, blocks
   completion until task-contract/evidence/diff-review checks are satisfied; always
   appends a read-only session-log draft (never stashes or discards uncommitted work).

### The core state model

- **`validation-state.json`** — `risk_mode`, `task`, `task_started_at`, `checks`,
  `attempts` (retry/circuit-breaker log), `command_outcomes` (bounded, redacted
  post-execution records), `assumptions`/`open_questions`, `mode_overrides`.
- **`state/task-queue.md`** — human-readable multi-task tracking, blockers, worktree
  isolation for 2+-task autopilot runs.
- **`state/evidence-ledger.json`** — per-claim evidence; only entries with
  `recorded_by="hook"` and a real `exit_code == 0` can satisfy a STRICT-mode
  completion claim (see §10 for the honest limits of this).

## 3. Update procedure

```bash
cd ~/projects/ai-projects/claude-toolkit
git checkout -b feature/<topic>              # never edit on init directly
# ... make your change ...
bash scripts/harness_check.sh "$PWD"         # must PASS before installing
bash install.sh                               # idempotent; copies skills/hooks/
                                               # scripts/agents/rules/policies/
                                               # mistakes/templates/commands into
                                               # ~/.claude — never touches settings.json,
                                               # secrets, or per-project state
git add <files> && git commit -m "..."
git push -u origin feature/<topic>
gh pr create --base init ...                  # branch → PR → merge, never direct to init
```
After merging, re-run `install.sh` from `init` if you didn't already sync from your
feature branch — `harness_check.sh` on the live copy is the fastest sanity check.

## 4. Rollback procedure

Every install this project's history performed took a timestamped backup first
(`~/.claude/backups/<name>-<timestamp>/`). To roll back a specific file:
```bash
ls ~/.claude/backups/                                    # find the right backup
cp ~/.claude/backups/<backup-dir>/<path>/<file> ~/.claude/<path>/<file>
bash scripts/harness_check.sh "$PWD"                      # confirm restored state
```
`install.sh` never deletes — every rollback is a plain file copy back over the live
version. If no backup was taken before a change (e.g. a quick manual edit), use
`git log`/`git show <commit>:<path>` on the source repo to recover the prior version,
then re-run `install.sh`.

## 5. Skill architecture

Four distinct kinds of on-demand content, each with a different job:

| Kind | Where | Loaded | Job |
|---|---|---|---|
| **Skill** | `skills/<name>/SKILL.md` | On demand, by description match | A reusable procedure — a multi-step workflow (e.g. `verified-ship`, `task-rigor`, `cloud-audit`). Can carry `disable-model-invocation: true` for skills that should only run when explicitly invoked (e.g. `/ship`). |
| **Subagent** | `agents/<name>.md` | On demand, via the `Agent` tool | An isolated specialist reviewer with its own tool restrictions (usually `Read, Grep, Glob, Bash`, read-only) and a fixed output contract (GO/NO-GO, confidence-labeled findings). Runs in a fresh context — no bias from the main conversation. |
| **Command** | `.claude/commands/<name>.md` | User types `/<name>` | A short, explicit trigger that routes to the right skill/agent — a slash-command shortcut, not new logic of its own. |
| **Rule** | `rules/<name>.md` | **Always**, every turn | Narrowly-scoped, short, behavior-defining. The only layer that costs context on every single prompt — kept small on purpose (see §6). |

**Routing** is description-based, not code-based: three tables are the actual source
of truth for "which skill/agent handles this" —
`policies/skill-routing.md` (skills), `policies/subagent-routing.md` (all subagents),
`.claude/commands/review.md` (the `/review` reviewer dispatch table specifically).
There is no separate dispatcher script; the orchestrating Claude reads these
tables/descriptions and picks the best match.

**A new agent/skill is not dispatchable by name until a fresh session starts** — the
available-subagent-type list is fixed for a session's lifetime (confirmed live,
`FINAL_HARNESS_EVALUATION.md` / `FINAL_CLOSURE_VERIFICATION.md`). This is a platform
characteristic, not a harness bug — always verify a new agent in a fresh session
before relying on it.

## 6. Rule hierarchy

Seven layers, from broadest/rarest-changing to narrowest/most-current:

1. **Global `CLAUDE.md`** (`~/.claude/CLAUDE.md`, 38 lines) — only stable behavior
   that applies everywhere; pointer-based (points at the rule files below, doesn't
   restate their content).
2. **Project `CLAUDE.md`** (`<project>/.claude/CLAUDE.md`) — project-specific
   architecture, commands, conventions. Only loaded when that project is active.
3. **Rules** (`rules/*.md`, always-loaded, currently 357 lines total —
   `behavior.md` 80, `cloud.md` 43, `security.md` 29, `troubleshooting.md` 167).
   Narrowly scoped where possible: `cloud.md` carries `paths:` frontmatter limiting it
   to Terraform/IaC/YAML/Dockerfile work (whether the platform currently *enforces*
   that path-scoping was not independently re-verified as of the last review — treat
   as **UNKNOWN**, not confirmed, until checked against current docs).
4. **Skills** (§5) — reusable procedures, loaded only when relevant.
5. **Subagents** (§5) — isolated investigation/specialist work.
6. **Task state** (`validation-state.json`, `task-queue.md`) — current objective,
   progress, failures, next step. Query it in one call with
   `python3 scripts/validation_state.py awareness <root>` instead of re-reading three
   files by hand.
7. **Evidence** (`evidence-ledger.json`) — test results, file references, commands,
   sources. Query with `python3 scripts/validation_state.py completion-checklist <root>`
   before claiming a task done.

**Why it's kept small at layers 1-3 specifically:** this harness's own history
(`deploy-states/harness-setup.md`) records a deliberate 2026-07-01 cut from ~360 to
107 always-loaded lines after finding that adherence drops once always-loaded context
exceeds roughly 200 lines. The current 357-line total is above that historical figure
again (mainly `troubleshooting.md`, added and protected later) — reviewed twice since
(`CONTEXT_ENGINEERING_REPORT.md`, `FINAL_CLOSURE_VERIFICATION.md`) and kept
deliberately, not by oversight: its content is a detailed, actively-used workflow, not
filler. Any future addition to layers 1-3 should ask first whether it belongs at layer
4/5 instead (a skill/agent), since that's the layer designed to carry detail without
cost.

## 7. How to add a new skill (or agent) safely

1. Check the three routing tables (§5) — does something already cover this? Don't
   add a duplicate (the skills audit reviewed 164 external candidates and imported
   zero for exactly this reason — see `SKILLS_IMPLEMENTATION_REPORT.md`).
2. Write it locally, copying an existing skill/agent's exact structure (frontmatter
   shape, output format, tool restrictions). Default to read-only tools
   (`Read, Grep, Glob, Bash`) unless write access is the explicit point.
3. Add it to the relevant routing table.
4. `bash scripts/harness_check.sh "$PWD"` — must still PASS.
5. `bash install.sh` to sync.
6. **Start a new session**, then dispatch it against a small, deliberately-flawed test
   fixture before trusting it on real work (see `FINAL_CLOSURE_VERIFICATION.md` §1-2
   for the exact methodology: plant 2-4 known issues, confirm the agent catches them
   and follows its output contract).
7. If adapting an *external* skill: individually inspect its `SKILL.md`, scripts,
   dependencies, permissions, network behavior, and maintenance status first — never
   install a whole package/repo sight-unseen.

## 8. How to debug problems

- **A hook isn't firing as expected** → check `~/.claude/settings.json`'s `hooks`
  block first (wired to the right event/matcher?), then
  `~/.claude/logs/prompt-audit.log` / `~/.claude/logs/tool-failures.log` for what
  actually ran.
- **`final_gate` keeps blocking / won't clear** →
  `python3 scripts/validation_state.py show <root>` then
  `python3 scripts/final_gate.py <root>` — it always prints every blocking reason by
  name, never a bare fail. If the reasons are stale/irrelevant (e.g. a shared `cwd`
  carrying history from an unrelated task), that's a known, documented pattern
  (`mistakes.yaml` M008) — never fabricate the missing checks to force it green;
  either genuinely satisfy them or use `force_set_risk_mode()` with a real
  reason+evidence (it will legitimately refuse if an unsafe signal is present).
- **A command I expected to be blocked wasn't, or vice versa** →
  `python3 scripts/command_policy.py --test` first (whole suite still 143/143?), then
  `python3 -c "import command_policy as cp; print(cp.classify('<cmd>', 'project-autopilot'))"`
  to see the exact classification.
- **Lost track of where a long task stands** →
  `python3 scripts/validation_state.py awareness <root>` (one aggregated view: task,
  phase, open failure sequences, assumptions, evidence count, whether human input is
  required) or `completion-checklist <root>` before claiming something done.
- **A newly added skill/agent isn't showing up** → start a new session (§5, §7).
- **Working tree looks dirty right after running task-rigor/task-queue tooling** →
  check `.claude/state/`/`.claude/validation-state.json` specifically — these are
  gitignored harness bookkeeping, not real changes (`worktree_isolate.is_clean()` and
  `changed_files_fence.py` already exclude them from their own checks; if you see them
  flagged somewhere else, that's the one known inconsistency in §9).

## 9. Known limitations

Carried forward honestly from `HARNESS_RELIABILITY_IMPLEMENTATION.md` §6,
`PHASE1_FINAL_CHECK.md`, and `FINAL_CLOSURE_VERIFICATION.md` — none of these are
secret or newly discovered, all are accepted trade-offs for a personal, non-adversarial
harness, not defects hiding from view:

- **Skills/agents are prompt-driven.** No hook forces a skill to actually call its own
  retry-tracking helpers or follow its documented workflow — the model has to choose
  to. Enforcement only exists where it's real code (hooks/scripts).
- **Evidence trust assumes a non-adversarial process.** `is_trusted()` closes
  accidental/casual fabrication through normal usage (the CLI, the real hooks) — it is
  **not cryptographic tamper-proofing**. A process with unrestricted local file/Python
  execution could hand-edit `evidence-ledger.json` directly or call
  `record_hook_evidence()` outside its intended call sites. Same class of limit every
  fail-safe hook here already accepts.
- **No max-edits / max-wall-clock / max-tool-call ceiling exists.** Only retry count
  (circuit breaker) and cloud-deploy cost are bounded.
- **`worktree_isolate.py`'s `.claude/` exclusion is broader** than
  `changed_files_fence.py`'s narrower one — inconsistent but not unsafe (see §8's last
  bullet).
- **`commands_run`/`command_outcomes` are capped (200 entries), not archived** — older
  entries are dropped, not preserved elsewhere, if deep history beyond the cap is ever
  needed.
- **A newly added local agent needs a fresh session** to become dispatchable by name
  (§5, §7) — not fixable from this repo, a platform session-boundary characteristic.
- **`cloud.md`'s path-scoped frontmatter's actual enforcement is UNKNOWN** — documented
  as intended but not independently re-verified against current platform behavior.

See `HARNESS_BACKLOG.md` for the non-blocking hardening/optimization ideas that follow
from these — none are scheduled; they're there if a real need arises.

## 10. Design principles

1. **Enforcement lives in code; knowledge lives in prose, loaded on demand.** If a
   rule matters enough to always apply, it should be a hook, not a paragraph everyone
   has to remember to follow.
2. **Never fabricate a check to satisfy a gate.** A gate that can be satisfied by
   lying about it isn't a gate — `evidence_ledger.is_trusted()`'s entire design (never
   trust a stored boolean, always recompute from raw fields) exists because of this
   principle, discovered the hard way (audit Test 10b).
3. **Fail-safe, not fail-blind.** Every hook here follows the same convention: an
   internal error defaults to `allow`/no-op (never becomes the reason something
   dangerous slips through by *failing closed into a worse state*), while the actual
   safety checks (secret files, destructive commands) are hard, unconditional blocks
   with no such escape hatch.
4. **Archive, never silently delete.** Stale state, superseded risk-mode overrides,
   old task-queues — all get a timestamped copy before being reset. Nothing in this
   harness's own state management destroys history to make a check pass.
5. **Additive, not duplicative.** New state fields extend existing structures
   (`validation-state.json` gained `attempts`, `assumptions`, `command_outcomes` over
   time) rather than spawning parallel files that could drift out of sync.
   `awareness_snapshot()` is a read-time aggregator over existing state, not a second
   copy of it.
6. **Prefer the simplest safe option over a bigger redesign.** Demonstrated repeatedly:
   0 of 164 external skills imported when existing coverage was already stronger; a
   circuit-breaker reuses the same fingerprint machinery for both raw commands and
   skill-level phases instead of two parallel systems; a context-size review that
   found real duplication fixed it in place rather than restructuring the rule files.
7. **Evidence before completion, always.** Never say "done," "fixed," "verified," or
   "safe" without a matching, real check — this is the single idea `final_gate.py`,
   `completion_checklist()`, and every report in this repo's own writing style all
   share.
8. **State scoped to task and time, not assumed fresh.** A project directory is often
   shared across unrelated conversations — `task_id` and `task_started_at` exist
   specifically so evidence/state from one task can't silently satisfy another, and
   stale state gets archived rather than trusted by default.
